# Antigravity Kit — руководство пользователя

Практический разбор кита: как он «думает», как пройти полный цикл и как выжать максимум из каждого этапа. Краткий справочник — в [README.ru.md](../README.ru.md). English version: [GUIDE.md](GUIDE.md).

---

## 1. Ментальная модель

«Из коробки» AI-агент старательный, но недисциплинированный: начинает писать код, не разобравшись в задаче, заявляет «тесты проходят», не запустив их, и жжёт ваш контекст (и квоту) на мусоре. Кит оборачивает такого агента в **конвейер на контрактах**:

```
kit-plan  →  kit-work  →  kit  →  kit-review
 (форма)     (исполнение) (цикл)  (проверка)
```

У каждого этапа есть **Definition of Done**, который он обязан выполнить до начала следующего. Вокруг цикла — три вспомогательные системы:

- **Хуки** — защитная сетка, работающая автоматически (блокирует `rm -rf /`, подсказывает токен-экономные команды, продолжает длинные задачи).
- **MCP-серверы** — дополнительные «органы чувств» агента (структурированное рассуждение, структура кода без чтения файлов целиком, свежие доки библиотек).
- **Токен-оптимизация** — RTK и Headroom, которые снижают, во сколько токенов обходится та же работа.

Всё запоминать не нужно. Начните с четырёх основных команд; остальное включается по мере необходимости.

## 2. Установка за минуту

Кит распространяется через GitHub (не через npm). Два способа:

```bash
# Глобально — на все проекты этой машины, все поверхности (IDE, CLI, 2.0)
npx github:sipki-tech/antigravity-kit#main install --full

# В проект — ложится в ./.agents/, закоммитьте, чтобы кит был у всей команды
cd your-project
npx github:sipki-tech/antigravity-kit#main install --full --workspace
```

`--full` также ставит [RTK](https://github.com/rtk-ai/rtk) (сжатие терминала) и [Headroom](https://github.com/chopratejas/headroom) (сжатие блобов). Уберите флаг, если нужен только плагин.

Затем:

```bash
# добавить слэш-команды /kit-* в текущий проект
npx github:sipki-tech/antigravity-kit#main workflows

# проверить, что всё встало
npx github:sipki-tech/antigravity-kit#main verify            # глобальная установка
npx github:sipki-tech/antigravity-kit#main verify --workspace # установка в проект
```

Перезапустите Antigravity, чтобы он подхватил плагин. (`#main` заставляет npx взять последний коммит, а не кэш.)

## 3. Первый цикл — разбор на примере

Пусть задача — **«добавить rate limiting на эндпоинт `/login`».** Вот полный цикл.

### Шаг 1 — План

Наберите слэш-команду или wake word:

```
/kit-plan add rate limiting to the /login endpoint
```

Агент создаёт **файл** плана `.agents/kit/plans/rate-limit-login.md` — а не просто текст в чате — с секциями:

- **TL;DR** — резюме в одну фразу.
- **Scope** — в объёме: обработчик логина и его middleware; вне: остальные эндпоинты.
- **Steps** — чекбоксами (`- [ ] добавить token-bucket middleware`, `- [ ] подключить к /login`, …).
- **Risks** — например, «общее состояние между инстансами требует хранилища, а не памяти».
- **Completion criteria** — «после N попыток возвращается 429, проверяется `npm test rate-limit`».

Код пока не пишется. Почему файл? Потому что Antigravity обрезает контекст на длинных задачах — план, живущий только в разговоре, забывается через три промпта. Файл переживает обрезание.

### Шаг 2 — Работа

```
/kit-work
```

Исполнитель находит план, **перечитывает его перед каждым шагом** и делает ровно то, что там написано, — не больше. Отмечает `- [ ]` → `- [x]` только после верификации шага. Если реальность расходится с планом (скажем, хранилище для rate-limit не настроено), он фиксирует отклонение явно, а не импровизирует молча.

### Шаг 3 — Цикл

```
/kit-loop
```

Test-first: написать тест, падающий по правильной причине, сделать минимальную правку для прохождения, показать зелёный прогон. Каждое «работает» подкреплено выводом команды, который видно. Команды идут через `rtk`, чтобы вывод не затапливал контекст.

### Шаг 4 — Ревью

```
/kit-review
```

Ревьюер читает дифф по ханкам **и обязан реально запустить** тесты и линт проекта до вердикта — одобрять по одному чтению диффа запрещено. На выходе структурированный вердикт: блокеры (Blocking), рекомендации (Recommended) и выполненные проверки (Checks).

Это и есть цикл. Для мелкой правки можно сразу идти в `/kit-loop`; для всего, где больше пары движущихся частей, — начинать с `/kit-plan`.

Это **лёгкий цикл** — ежедневный драйвер. Для крупных high-stakes фич есть более тяжёлый **спек-режим** (раздел 4b) с формальными требованиями, дизайном и принудительными апрув-гейтами. Глубину выбираете вы.

## 4. Этапы подробно

| Когда вы… | Команда | Что даёт |
| --- | --- | --- |
| перед нетривиальной задачей | `kit-plan` | план-файл; кода нет до вашего одобрения |
| план одобрен | `kit-work` | дисциплинированное исполнение, отклонения фиксируются |
| пишете реализацию | `kit` / `kit-loop` | test-first циклы с видимыми доказательствами |
| готовитесь отгружать | `kit-review` | вердикт, подкреплённый выполненными тестами |
| видите AI-мусор в диффе | `kit-clean` | мёртвый код/шум удалён, поведение не изменилось |
| поймали баг | `kit-debug` | воспроизведение → гипотезы → минимальный фикс |
| цель на несколько сессий | `kit-goal` | чек-лист, переживающий рестарты |
| планируете `/teamwork-preview` | `kit-teamwork` | полный бриф до траты квоты |
| строите сложную/high-stakes фичу | `kit-spec` | фазовый спек-пайплайн с принудительными гейтами (раздел 4b) |

Ещё два скилла подгружаются автоматически по необходимости и вызывать по имени их не нужно:

- **kit-seq-thinking** — учит агента, когда браться за sequential-thinking (многошаговое планирование, неочевидные баги) и держать бюджет мыслей, чтобы не переусердствовать на тривиальных правках.
- **kit-token-hygiene** — привычки экономии токенов: префикс `rtk`, чтение диапазонов вместо файлов целиком, никаких сырых логов.

### Два способа вызвать любой этап

- **Слэш-команда** — `/kit-plan`. Надёжно, с автодополнением, композируется. Требует установленных workflows (`… workflows`).
- **Wake word** — просто написать `kit-plan …` в промпте. Удобно, но эвристика. Хук игнорирует текст, похожий на идентификаторы (`kit_helper.mjs`, `src/kit-plan/`), а голое `kit` срабатывает только первым словом.

## 4b. Спек-режим — для сложных фич

Лёгкого цикла хватает, пока фича не стала большой и рискованной настолько, что ошибка в требованиях дорого обходится. **Спек-режим** (`/kit-spec`) добавляет три фронт-фазы и навязывает апрув-гейты детерминированным движком состояния:

```
explore → requirements → design → task-plan → implementation → review → done
          [approve]      [approve] [approve]  [approve]       [approve] [approve]
```

Последние три фазы **переиспользуют уже знакомые скиллы** — task-plan это `kit-plan`, implementation это `kit-work` + `kit-loop`, review это `kit-review`. Спек-режим добавляет только фронт (explore, формальные требования WHEN/SHALL, дизайн-ADR) и enforcement.

**Как это идёт.** Один вход — `/kit-spec add billing` — и оркестратор ведёт последовательность. На каждой фазе он пишет артефакт в `.agents/kit/pipeline/<feature>/`, затем **останавливается на гейте** и ждёт твоего апрува. Одобрил — двинулся дальше. Гейты настоящие: маленький Node-движок (`pipeline.mjs`) держит фазу и **не даёт продвинуться без зарегистрированного артефакта** — агент не проскочит фазу и не объявит готовым раньше времени. Stop-хук подстрахует: уйдёшь в сторону с висящим пайплайном — напомнит, какая фаза ждёт.

**Почему движок, а не просто markdown?** Потому что в спек-подходе весь смысл в том, что гейты *принудительны*, а не пожелания — тот же принцип, что `kit-review` с обязательными прогнанными тестами. Плюс движок переживает обрезание контекста и чисто возобновляется.

**Когда применять:** неясные требования, архитектурные решения, всё где недопонимание дорого. **Когда нет:** мелкие правки и баг-фиксы — это лёгкий цикл, а спек-режим намеренно тяжёлый.

```bash
/kit-spec <feature>        # старт / продолжение
/kit-spec-status           # где я?
/kit-spec-approve          # продвинуть после проверки артефакта
```

Требования, записанные здесь (`R1`, `R2`, …), прослеживаются насквозь: `kit-review` проверяет, что каждое требование связано и с изменением, и с тестом.

### kit vs SDD

Спек-пайплайн вдохновлён [sipki-tech/sdd](https://github.com/sipki-tech/sdd) — зрелым универсальным спек-скиллом. Разница:

- **kit** нативен для Antigravity и самодостаточен — пайплайн работает как workflows плагина, Stop-хук показывает состояние, всё в `.agents/`, настроено под Gemini. Ставить что-то ещё не нужно.
- **SDD** универсален — работает на 40+ агентах (Claude Code, Cursor, Codex, Copilot…) ради переносимости.

Выбирай по потребности: остаёшься в Antigravity → спек-режим кита; нужен тот же пайплайн в разных инструментах → SDD. Они сиблинги, ни один не требует другого.

## 5. Отладка через kit-debug

`kit-debug` существует, потому что агенты обожают «фиксить наугад». Контракт навязывает дисциплину:

1. **Сначала воспроизвести** одной командой. Нет воспроизведения → сбор информации *и есть* задача; не чините вслепую.
2. **Гипотезы** через sequential-thinking, когда причина неочевидна, — по одной за раз, каждая с опровергающей проверкой.
3. **Проверить механизм** — показать конкретную строку/состояние, вызывающие сбой, до правок кода.
4. **Минимальный фикс** — минимум кода, убирающий причину; посторонние чистки — в отдельную задачу.
5. **Доказать** — воспроизведение падает до и проходит после; прогнать весь набор тестов.

## 6. Длинные цели через kit-goal

Для работы на несколько сессий `kit-goal` превращает бриф в чек-лист `.agents/kit-goal.md`:

```markdown
# kit-goal: migrate auth to JWT

- [ ] access-токены выдаются при логине — проверка: `npm test auth/login`
- [ ] refresh-флоу работает — проверка: `npm test auth/refresh`
- [ ] старый код сессий удалён — проверка: `grep -r session src/ | wc -l` == 0
```

**Хук goal-continuation** читает этот файл на каждом событии Stop. Пока есть неотмеченные пункты, он просит агента продолжать, а не останавливаться раньше времени. Когда все галочки стоят — молчит. Пункты отмечаются только после запуска их команды-проверки.

## 7. Токен-оптимизация

Боль №1 в Antigravity — квота, сгорающая за 10–15 минут. Кит бьёт по ней с трёх сторон:

1. **RTK** (вывод терминала). С `--with-rtk` кит ставит бинарник и запускает `rtk init --agent antigravity`, регистрируя собственный rewrite-хук rtk — каждый `git status`, `npm test` и т.д. сжимается на 60–90% автоматически. Хук `rtk-enforcer` кита — вежливый фолбэк, подсказывающий префикс; он молчит, когда нативный хук rtk активен.
2. **Headroom** (всё остальное — логи, файлы, RAG). Зарегистрирован как MCP-сервер; агент вызывает `headroom_compress` на крупных блобах. Автовключается при наличии CLI `headroom` (`--with-headroom` его ставит).
3. **tree_sitter** (полное избегание чтения). Агент спрашивает у AST нужный символ вместо чтения файла на 2000 строк — мусор, не попавший в контекст, не нужно сжимать.

**Обходные пути**, когда нужен сырой вывод:

- `KIT_RAW=1 <команда>` — одна команда без rtk-подсказки.
- `KIT_RTK_ENFORCE=off` — отключить подсказку на сессию.

Замерить экономию на своём репозитории — по [docs/token-benchmark.md](token-benchmark.md).

## 8. Подготовка к /teamwork-preview

`/teamwork-preview` (Antigravity 2.0, план Ultra) поднимает команду из нескольких агентов, но **один запуск может сжечь недельную квоту** и проваливается на размытых брифах. Поэтому кит заставляет подготовиться:

```
/kit-teamwork build the analytics dashboard
```

Сначала проверяется применимость — связные рефакторинги и последовательные задачи возвращаются к `kit-plan` + `kit-work`, где дешевле и надёжнее. Если teamwork действительно подходит (широкая параллелизуемая работа), пишется `.agents/kit/teamwork-brief.md` с критериями успеха (каждый с командой проверки), явными non-goals и **непересекающимися lanes** — потому что параллельные агенты молча затирают один файл, если двое его трогают.

Скилл сам запуск не делает; решение о квоте — за вами. Даже если вы просто упомянули `/teamwork-preview` вскользь, wake-word-хук напомнит сначала подготовить бриф.

## 9. Настройка для команды

Чтобы вся команда получила одного дисциплинированного агента без настройки на каждого:

```bash
cd your-repo
npx github:sipki-tech/antigravity-kit#main install --full --workspace
npx github:sipki-tech/antigravity-kit#main workflows
git add .agents && git commit -m "add antigravity-kit"
```

Теперь `.agents/plugins/antigravity-kit/` и `.agents/workflows/` в репозитории. Любой, кто откроет проект в Antigravity, автоматически получит плагин, хуки, скиллы и команды `/kit-*` — одни правила, одни ревью-гейты, одна токен-дисциплина на всю команду.

Что коммитить, а что игнорировать: общий кит (`.agents/plugins/`, `.agents/workflows/`, `.agents/mcp_config.json`) — коммитить, локальные рабочие артефакты — нет. Рекомендуемый сниппет для `.gitignore` проекта:

```gitignore
# antigravity-kit: локальные рабочие артефакты (планы, пайплайны, цели)
.agents/kit/
.agents/kit-goal.md
```

Если проект уже игнорирует `.agents/` целиком, наоборот раз-игнорируйте общие части:

```gitignore
.agents/*
!.agents/plugins/
!.agents/workflows/
!.agents/mcp_config.json
```

Файлы планов (`.agents/kit/plans/`) — полезные артефакты для PR; если хотите их коммитить — уберите строку `.agents/kit/`; `.agents/kit-goal.md` в любом случае обычно временный.

## 10. Безопасность и предохранители

Кит спроектирован так, чтобы никогда не ломать сессию:

- **Все хуки fail-open.** Любая внутренняя ошибка разрешается в «allow» — баг в хуке не заблокирует работу.
- **danger-guard** блокирует только уверенные опасности: `rm -rf` вне workspace, force-push в main/master, чтение `.env*`/ключей. Всё остальное проходит.
- **Отсутствующие инструменты деградируют мягко.** Нет бинарника rtk? Команды идут как есть. Нет headroom? Его MCP-запись остаётся выключенной.
- **`KIT_SAFE_MODE`** зарезервирован как глобальный рубильник для будущих функций хуков.

Профили прав (`--permission-profile safe|balanced|full|none`, по умолчанию `safe`) настраивают, что агент может автоодобрять; сейчас enforcement идёт через rules и хуки, пока схема настроек Antigravity не стабилизируется.

## 11. Траблшутинг

**`npx antigravity-kit` ставит не то.** Нескоуповое имя в npm — чужой пакет. Всегда используйте GitHub-форму: `npx github:sipki-tech/antigravity-kit#main`.

**Изменения не подхватываются.** npx кэширует установки с GitHub. Добавьте `#main`, чтобы взять последний коммит, или выполните `npm cache clean --force`.

**`verify` падает после установки в проект.** Используйте `verify --workspace` — простой `verify` проверяет глобальный путь `~/.gemini`.

**Папка плагина пуста / не там, где ждал.** `--workspace` ставит в `<project>/.agents/plugins/`, а не в глобальный `~/.gemini/config/plugins/`. Скиллы живут *внутри* директории плагина, отдельной папки скиллов нет. И проверьте, что не запускали с `--dry-run` — он ничего не пишет.

**Wake words не срабатывают.** Предпочитайте слэш-команды (`/kit-plan`) — они надёжны. Wake words — эвристика и намеренно игнорируют код-подобный контекст.

**Субагенты (`kit-planner`, `kit-reviewer`) не появляются.** Кастомные субагенты могут требовать платный план Antigravity. Остальной кит работает независимо.

## 12. FAQ

**Обязательно проходить весь цикл каждый раз?** Нет. Тривиальные правки — сразу в `/kit-loop`; мелким фиксам план не нужен. Цикл — для работы, где дисциплина окупается.

**Работает вне Antigravity?** Скиллы — переносимый формат Agent Skills. `install --host claude-code` или `--host codex` копирует корпус скиллов туда. Хуки и манифест специфичны для Antigravity — детали в [docs/portability.md](portability.md).

**Не замедлит ли меня?** Хуки добавляют миллисекунды и fail-open. Экономия токенов и эффект «нет переделок, потому что план был ясен» с лихвой окупают структуру.

**Можно кастомизировать правила?** Да — правьте `rules/*.md` в плагине (стиль вывода, политика безопасности, конвенции workflow). При установке в проект они лежат в `.agents/plugins/antigravity-kit/rules/`.
